Description

This patient summary document example highlights some factors not often present in other examples.

Tone of Voice of an Empowered Patient

The narratives in the document are written in first person, as the patient.

This emphasises the fact that in many cases it is the patient who has the most complete and up-to-date information on their health, on their circumstances, and on their goals and wishes.

We should acknowledge that when working with patient summaries.

Patient-Reported Data

The summary is authored by the patient. Although it is composed of data elements from authoritative healthcare organizations, the patient remains the author of the data. Both the document and all the individual resources have been tagged with meta.security flags as patient reported to highlight this fact, and Provenance resources provide additional details.

In the EU, the legislation for the European Health Data Space (EHDS) mandates EHRs to clearly indicate which data is reported by the patient. The tags and provenance records should facilitate that.

Narratives for Humans, Resources for Machines

The example includes many hand-crafted human-readable narratives. For instance, all sections of the Composition have narratives.

These narratives have the .text.status set to additional. Any IPS viewer should pay attention to this and render the narratives.

Many resources in the example have automatically generated narratives. These narratives have the .text.status set to generated. It is totally fine for an IPS viewer to ignore these narratives if the viewer displays the contents of the resource anyway.

Advance Directives Including Data Use

The summary includes advance directives as Consent resources. Some of these are about wishes regarding treatment, but there are also some regarding the use of data in this document.

Please pay attention to these directives and to what they say about how you may use this document and the data within it!

Test Data

This document is flagged as test data in the Bundle's .meta.security element. The security element of the Bundle also tells that the dodument is supposed to be used in system development and training.

This information is only present in the Bundle element, but the interpretation should be that it applies to all of the individual resources within the Bundle too.

Parallel Codes

Many resources are coded with multiple parallel coding systems. Some are from original sources (used within the Finnish healthcare system), but many other codes have been added for international interoperability.

Declaration of Supported Profiles

The document and many individual resources declare support for multiple profiles in meta.profile tags.

The document claims to conform to the International Patient Summary (IPS), to the European Patient Summary (EPS), to the International Patient Access (IPA), to the Finnish FHIR Base profiles, and to several other profiles that those specifications reference.
